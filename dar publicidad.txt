ID 1-1 - Iniciar Sesion
Descripción: Este caso de uso se enfocaría en el inicio de sesión al sistema de parte de un usuario registrado, permitiendo acceso al resto de casos de uso.
Actores: Administrador del sistema, operador de radio, vendedor, productor publicitario.
Prioridad: Alta
Riesgo: Alta (esencial para la seguridad y la integridad de los datos).
Disparador: Se contrata un nuevo empleado, un empleado cambia de rol o se requiere modificar los permisos de acceso.
Requerimientos no funcionales:
La interfaz debe ser intuitiva y permitir el registro de un nuevo usuario en menos de 2 minutos.
El sistema debe validar datos obligatorios (nombre, contacto, teléfono o correo).
La información debe almacenarse de forma segura en la base de datos, cumpliendo con estándares de encriptación.
El sistema debe permitir la auditoría de cambios en los permisos de usuario.
Pre-Condiciones: Debe existir una credencial de usuario administrador precargada al inicio del sistema.
Post-Condiciones: El usuario accede al sistema y puede realizar todos los flujos que tenga habilitados.

flujo normal
Inicio: El usuario accede al sistema, es redirigido por el sistema al menú “Iniciar sesión”.
El sistema muestra un formulario con las credenciales necesarias para un inicio válido.
El usuario completa los campos obligatorios.
El sistema valida los datos ingresados.
Si la validación es exitosa, el sistema confirma el inicio de sesión.
El sistema muestra un mensaje de confirmación:
"Inicio de sesión exitoso."
Fin: El sistema presenta un menú que permite acceder al resto de flujos del sistema a los que este usuario está validado para utilizar.

flujo alternativo: credenciales incorrectas
Inicio: El sistema detecta credenciales erróneas por parte del usuario que desea ingresar.
El sistema avisa al usuario del error y pide intentarlo nuevamente.
Si valida datos correctos, regresa al flujo normal. De lo contrario, reinicia este flujo alternativo.

id 2-1 - registrar usuario
Descripción: Permite a un administrador crear credenciales para nuevos usuarios, incluyendo subtipos de usuarios o administradores adicionales.
Actores: Administrador del sistema.
Prioridad: Alta.
Riesgo: Alta (esencial para la seguridad y la integridad de los datos).
Disparador: Se requiere registrar un nuevo usuario con acceso a una determinada parte del sistema.
Requerimientos no funcionales:
La interfaz debe ser intuitiva, permitiendo el registro de un nuevo usuario en menos de 2 minutos.
El sistema debe validar datos obligatorios (nombre, contacto, teléfono o correo).
La información debe almacenarse de forma segura, cumpliendo con estándares de encriptación.
Pre-Condiciones: Debe existir una credencial de usuario administrador precargada al inicio del sistema.
Post-Condiciones: El nuevo usuario queda dado de alta, puede acceder al sistema y puede realizar los flujos habilitados según su rol.

flujo normal
Inicio: El administrador selecciona la opción "Registrar nuevo usuario" en el menú principal.
El sistema muestra un formulario con campos para nombre, contacto, teléfono, correo, rol y permisos.
El administrador completa los campos obligatorios y opcionales.
El sistema valida los datos ingresados.
Si la validación es exitosa, el administrador confirma el registro.
El sistema crea las credenciales y las almacena de forma segura.
El sistema muestra un mensaje de confirmación:
"Usuario registrado exitosamente."
Fin: El nuevo usuario está disponible para iniciar sesión en el sistema.

flujo alternativo
Inicio: El administrador intenta registrar un nuevo usuario con datos incompletos o inválidos.
El sistema detecta campos obligatorios faltantes o formatos incorrectos.
El sistema muestra un mensaje de error indicando los campos a corregir.
El administrador corrige los datos y retoma el flujo normal.
Fin: El usuario queda registrado si los datos son corregidos y validados.

id 2-2 editar usuarios
Descripción: Un administrador puede modificar las credenciales de otro usuario.
Actores: Administrador del sistema.
Prioridad: Alta.
Riesgo: Alta (esencial para la seguridad y la integridad de los datos).
Disparador: Un usuario cambia de rol, o se requiere modificar permisos de acceso.
Requerimientos no funcionales:
La interfaz debe permitir la edición de campos de forma intuitiva.
El sistema debe validar los datos modificados para asegurar su integridad.
Debe registrarse un historial de cambios para auditoría.
Pre-Condiciones: Debe existir una credencial de usuario administrador precargada al inicio del sistema que pueda realizar cambios a otras cuentas.
Post-Condiciones: La información del usuario se actualiza en la base de datos.

Inicio: El administrador selecciona la opción "Gestionar Usuarios" en el menú principal.
El administrador escoge de una lista de usuarios existentes el usuario a modificar.
Modifica la información deseada en el formulario.
El sistema valida los datos y solicita confirmación.
El sistema almacena los cambios en la base de datos.
Fin: Confirma la acción y el sistema actualiza la información del usuario en la base de datos.

flujo alternativo: datos incompletos o invalidos
Inicio: El administrador intenta modificar un usuario con datos incompletos o inválidos.
El sistema detecta campos obligatorios faltantes o formatos incorrectos.
El sistema muestra un mensaje de error:
"Faltan campos obligatorios o están en formatos inválidos."
Fin: Se reinicia el flujo normal desde el paso 3, solicitando nuevamente datos válidos.

flujo alternativo: cancelación por parte del actor
Inicio: El administrador selecciona la opción "Gestionar Usuarios" y luego el usuario a modificar o eliminar.
El sistema solicita confirmación antes de guardar los cambios o eliminar la cuenta.
El administrador cancela la acción.
El sistema muestra un mensaje:
"No se guardarán los cambios" o "La acción ha sido cancelada."
Fin: El sistema descarta los datos ingresados y la información del usuario permanece inalterada.

id 2-3 eliminar usuarios
Descripción: Un administrador puede eliminar las credenciales de otro usuario. No puede eliminarse a sí mismo, por lo que siempre debe quedar al menos una cuenta de administrador.
Actores: Administrador del sistema.
Prioridad: Alta.
Riesgo: Alta (esencial para la seguridad y la integridad de los datos).
Disparador: Se necesita eliminar una cuenta.
Requerimientos no funcionales:
Debe registrarse un historial de cambios para auditoría.
El sistema debe solicitar confirmación antes de eliminar una cuenta.
La eliminación debe ser lógica (marcar como eliminada) para mantener la integridad referencial.
Pre-Condiciones: Debe existir una credencial de usuario administrador precargada al inicio del sistema que pueda realizar esta acción, como una cuenta raíz del sistema.
El usuario debe haber iniciado sesión en el sistema con permisos de administrador.
Post-Condiciones: La cuenta se marca como eliminada lógicamente.

flujo normal
Inicio: El administrador selecciona la opción "Gestionar Usuarios" en el menú principal.
El sistema muestra una lista de usuarios existentes.
El administrador selecciona el usuario que desea eliminar.
El sistema solicita una confirmación antes de proceder con la eliminación:
"¿Está seguro de que desea eliminar este usuario? Esta acción no se puede deshacer."
El administrador confirma la eliminación.
El sistema marca el usuario como eliminado lógicamente en la base de datos.
El sistema muestra un mensaje de confirmación:
"El usuario se ha eliminado correctamente."
Fin: El usuario ya no es visible en las listas activas del sistema.

flujo alternativo: datos incompletos/invalidos
Inicio: El administrador intenta modificar o eliminar un usuario con datos incompletos o inválidos.
El sistema detecta campos obligatorios faltantes o formatos incorrectos.
El sistema muestra un mensaje de error:
"Faltan campos obligatorios o están en formatos inválidos."
El administrador completa los campos en un formato válido.
Fin: Se retoma el flujo normal, se confirma la acción y el sistema actualiza la información del usuario en la base de datos.

flujo alternativo: intento de autoeliminación
Inicio: El administrador intenta eliminarse a sí mismo.
El sistema detecta que es un intento de autoeliminación.
El sistema muestra un mensaje de error: "No puede eliminarse a sí mismo."
Fin: El flujo termina sin realizar cambios.

flujo alternativo: cancelación por parte del actor
Inicio: El administrador selecciona la opción "Gestionar Usuarios" y luego el usuario a modificar o eliminar.
El sistema solicita confirmación antes de guardar los cambios o eliminar la cuenta.
El administrador cancela la acción.
El sistema muestra un mensaje:
"No se guardarán los cambios" o "La acción ha sido cancelada."
Fin: El sistema descarta los datos ingresados y la información del usuario permanece inalterada.

id 3-1: registrar cliente nuevo
Descripción: Permite incorporar al sistema los datos de un nuevo cliente, registrando su información básica de contacto, razón social y rubro comercial, para futuras campañas publicitarias.
Actores: Vendedor / Administrador del sistema
Prioridad: Alta
Riesgo: Bajo — se trata de un proceso simple y frecuente.
Disparador: Un vendedor u administrador desea registrar un nuevo cliente interesado en contratar servicios publicitarios.
Requerimientos no funcionales:
La interfaz debe ser intuitiva y permitir el registro en menos de 2 minutos.
El sistema debe validar datos obligatorios (nombre, contacto, teléfono o correo).
La información debe almacenarse de forma segura en la base de datos.
Pre-Condiciones: El usuario debe haber iniciado sesión en el sistema con permisos de vendedor o administrador.
Post-Condiciones: El cliente queda registrado en la base de datos y disponible para asociar futuras campañas publicitarias.

Inicio: Actor selecciona la opción “Registrar nuevo cliente” en el menú principal.
Sistema muestra el formulario de alta de cliente con los campos requeridos:
Nombre o razón social
Rubro comercial
Datos de contacto (teléfono, correo electrónico, dirección, etc.)
Actor completa todos los campos obligatorios.
El sistema valida los datos ingresados (campos obligatorios y formato de correo/teléfono).
Si la validación es exitosa, el actor presiona el botón “Guardar”.
El sistema almacena la información en la base de datos de clientes.
El sistema muestra un mensaje de confirmación:
“El cliente se ha registrado exitosamente.”
Fin: El cliente queda disponible para futuras campañas publicitarias.

flujo alternativo: Inicio: Actor selecciona la opción “Registrar nuevo cliente” en el menú principal.
Sistema muestra el formulario de alta de cliente con los campos requeridos:
Nombre o razón social
Rubro comercial
Datos de contacto (teléfono, correo electrónico, dirección, etc.)
Actor completa todos los campos obligatorios.
El sistema valida los datos ingresados (campos obligatorios y formato de correo/teléfono).
El sistema detecta que falta un campo obligatorio o hay un formato incorrecto (por ejemplo, correo sin “@”).
El sistema muestra un mensaje de error indicando el campo a corregir.
Fin: El actor corrige los datos y repite el paso 6 del flujo normal.

id 4-1 crear campaña publicitaria
Descripción: Permite crear una nueva campaña asociada a un cliente, definiendo su tipo de medio (radio, televisión, redes, diario), duración, requerimientos del cliente, responsables de producción y estado de avance.
Actores: Productor publicitario / Administrador
Prioridad: Alta
Riesgo: Medio — involucra coordinación entre varias áreas y carga de materiales.
Disparador: Un cliente aprueba el inicio de una campaña o solicita la elaboración de una propuesta publicitaria.
Requerimientos no funcionales:
El sistema debe permitir adjuntar archivos (guiones, bocetos, audios, etc.) de manera rápida.
Las operaciones deben confirmarse antes de guardar los cambios.
Debe garantizarse la integridad de los datos mediante validaciones.
Pre-Condiciones: Debe existir un cliente previamente registrado.
El usuario debe tener permisos de productor o administrador.
Post-Condiciones: La campaña queda registrada con estado “En producción”, y sus datos disponibles para edición o seguimiento.

Inicio: Productor Publicitario/Administrador selecciona la opción “Crear nueva campaña” en el menú principal.
El sistema muestra el formulario de registro de campaña, con los siguientes campos:
Cliente asociado (seleccionable de una lista de clientes registrados) o cliente nuevo.
Duración (fechas de inicio y fin)
Requerimientos o descripción de la campaña
Responsables de producción
Estado de avance (por defecto: “En producción”)
El actor selecciona un cliente asociado, completa los datos requeridos y puede adjuntar archivos relacionados (guiones, bocetos, audios, imágenes, etc.).
El sistema valida que los campos obligatorios estén completos y que el cliente seleccionado exista.
El sistema solicita confirmación antes de guardar los cambios:
“¿Desea registrar esta nueva campaña publicitaria?”
El actor confirma la acción.
El sistema almacena la información en la base de datos y asocia la campaña al cliente seleccionado.
El sistema muestra un mensaje de éxito:
“La campaña se ha registrado correctamente con estado En producción.”
Fin: La campaña queda disponible para edición, seguimiento y asignación de recursos.

flujo alternativo: asociar campaña a cliente nuevo
Inicio: El sistema detecta que faltan campos obligatorios (por ejemplo, cliente no seleccionado o fechas inválidas).
El sistema muestra un mensaje indicando los errores encontrados.
Fin: El actor corrige la información y vuelve al paso 6 del flujo normal.

flujo alternativo: datos incompletos/invalidos

Inicio: El sistema detecta que faltan campos obligatorios (por ejemplo, cliente no seleccionado o fechas inválidas).
El sistema muestra un mensaje indicando los errores encontrados.
Fin: El actor corrige la información y vuelve al paso 6 del flujo normal.

flujo alternativo: cancelación por parte del actor
Inicio: El sistema solicita confirmación antes de guardar los cambios:
“¿Desea registrar esta nueva campaña publicitaria?”
El actor selecciona “Cancelar”.
Fin: El sistema descarta los datos ingresados y vuelve a la pantalla principal sin guardar cambios.

id 4-4 generar reporte de campaña
Descripción: Este caso de uso facilita la creación de informes personalizados sobre el rendimiento de las campañas publicitarias, incluyendo datos como la duración, los medios utilizados, los costos asociados y los resultados obtenidos (por ejemplo, alcance, interacciones si se integra con plataformas digitales).
Actores: Administrador, Productor Publicitario.
Prioridad: Media a Alta (importante para la toma de decisiones y la retroalimentación al cliente).
Riesgo: Bajo — la generación de reportes incorrectos puede llevar a decisiones erróneas.
Disparador: Un usuario necesita analizar el rendimiento de una campaña o presentar resultados a un cliente.
Requerimientos no funcionales:
La interfaz debe mostrar una vista clara del horario y la disponibilidad de espacios.
El sistema debe evitar la duplicación de anuncios en un mismo bloque horario.
El registro de tandas debe realizarse con respuesta inmediata del sistema (< 3 segundos).
Pre-Condiciones: Las campañas deben haber finalizado o estar en curso con datos registrados en el sistema. El usuario debe tener permisos de Administrador, Productor Publicitario, Vendedor.
Post-Condiciones: El reporte se genera con la información solicitada y está disponible para visualización o exportación.


flujo alternativo: campaña con datos insuficientes

Inicio: El usuario selecciona la campaña para generar el reporte.
El sistema verifica la existencia de datos asociados.
Detecta que la campaña no tiene información suficiente.
El sistema muestra el mensaje:
"La campaña seleccionada no posee datos disponibles para generar el reporte."
El usuario puede:
Seleccionar otra campaña
cancelar la operación.
Si selecciona otra campaña, el sistema vuelve al paso de selección de campaña.
Fin: El usuario procede con otra campaña o finaliza la operación.

flujo alternativo: error al generar reporte

Inicio: El usuario confirma la generación del reporte.
El sistema comienza a recopilar la información.
Ocurre un error durante el procesamiento de datos.
El sistema muestra el mensaje:
"No se pudo generar el reporte. Intente nuevamente o contacte al administrador."
El sistema registra el error en el log.
El usuario puede:
Reintentar la operación
Salir del módulo de reportes.
Si lo intenta nuevamente y es exitoso, continúa desde la visualización del reporte.
Fin: El reporte se genera o el usuario sale del módulo.
